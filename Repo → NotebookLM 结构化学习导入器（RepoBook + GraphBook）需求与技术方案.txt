Repo → NotebookLM 结构化学习导入器（RepoBook + GraphBook）需求与技术方案
1. 背景与目标

NotebookLM 支持将网页链接、YouTube、文档等作为 Source 作为上下文进行问答与学习，并能生成音频总结。但当 Source 是 GitHub 仓库时，如果仅提供仓库主页 URL，NotebookLM 通常只能获取主页内容（README/首页信息），无法自动获取仓库内所有目录与文件内容。手工逐个进入目录复制每个文件链接极其低效，而且会把原本树状结构的项目“扁平化”为一堆无层级的链接，不利于学习、检索、理解模块关系。

本项目目标是构建一个工具，将任意 GitHub 仓库“编译”为 NotebookLM 友好的结构化学习材料，做到：

全量获取（可选择性）：自动抓取仓库文件树与核心文本内容，不再手工复制每个文件链接。

结构保真：保留原仓库目录层级、主次结构与定位信息（路径）。

架构增强：除了代码正文，还提供一份“架构地图/依赖图谱”文档，提升 NotebookLM 回答的架构思维与导航能力。

双轨制 Source：将“代码正文（RepoBook）”与“结构/依赖（GraphBook）”作为两个独立 Source 上传到 NotebookLM（通过 notebooklm-py 等非官方 API/自动化工具）。

可增量更新：仓库更新时只重新处理变更文件与受影响关系，避免每次全量重算。

2. 核心产物（Artifacts）

工具对一个仓库输出三类产物：

RepoBook（代码与文档正文）

形式：一个或多个 Markdown 文件（建议按目录拆章）

内容：README、docs、核心源码文件内容（必要时截断），并保留路径/标题/目录树

目的：为 NotebookLM 提供“证据与细节”

GraphBook（结构与依赖架构地图）

形式：一个 Markdown 文件

内容：目录职责（每个目录做什么）、入口/关键路径推断、Import Graph（谁 import 谁）、以及后续可扩展的符号级关系（谁定义了谁、谁引用了谁、calls 置信度等）

目的：为 NotebookLM 提供“架构先验与导航地图”，让其回答更具结构性

manifest.json / graph.json（机器可读索引）

manifest.json：文件树、sha/hash、大小、语言、是否文本等，用于增量更新

graph.json：依赖边（imports 等）、目录职责与信号，供 GraphBook 生成与后续扩展

最终会通过 notebooklm-py（NotebookLM 非官方 API/CLI）将 RepoBook 与 GraphBook 作为两个 Sources 上传到同一个 Notebook 中。

3. 分阶段路线图（从易到难）
Phase 0：仓库抓取与结构保真（MVP 基础）

输入：GitHub repo URL（可选分支/commit）

动作：

获取仓库文件树（路径、大小、文件类型）

过滤二进制与不需要目录（node_modules、dist、.git、build 等）

下载文本文件内容（可限制最大文件大小）

生成 manifest.json

生成 RepoBook（至少包含 README + 目录树 + 核心文件章节）

Phase 1：Import Graph（Python + JS/TS）+ 目录职责（Architecture Map）

目标：先把“骨架结构”与“模块依赖”做扎实，立刻提升 NotebookLM 架构感。

Import Graph（模块/文件级依赖）：

Python：解析 import x、from x import y、相对导入 from . import ...

JS/TS：解析 import ... from、require()、import()（先保守）

Resolve 内部模块：将 import 目标尽量映射到 repo 内真实文件路径；无法映射的标为 external

输出边类型：imports（from_file -> to_file），并带 evidence（行号/片段）与 confidence（通常 high）

目录职责（Directory Responsibility）：

不依赖 LLM 也能做出稳定产物，基于启发式信号：

文件类型占比（py/ts/md/config）

典型目录命名（routes/controllers/services/models/utils/core/tests/docs/scripts）

与入口文件的距离（入口主干路径更核心）

import 图谱中心性（入度高通常是基础层/公共库）

输出每个目录的 role（一句话）+ why（2-3 个信号）

生成 GraphBook.md：

项目总览（来自 README 与入口检测）

目录职责地图（树状缩进）

核心依赖关系（Top N 节点/目录）

分层推断（基础层/业务层/接口层）

关系明细（imports 按 from 分组）

Phase 2：符号级（Symbol-level）索引与引用（逐步增强）

目标：从文件级依赖走向类/函数级导航，让 NotebookLM 能回答“这个函数在哪定义”“哪个模块使用了它”。

Symbol Index：

Python：AST 提取顶层 FunctionDef/ClassDef/Assign（可包含方法）

TS/JS：TypeScript compiler API 提取 export/function/class/const

输出：defines、exports、reexports

Symbol Reference（保守引用）：

Python：AST 扫描 Name/Attribute，结合 import resolve 做简单符号归属推断

TS：Identifier/PropertyAccess + 类型/导入信息做 resolve（可先简化）

输出边类型：references（from_file or from_symbol -> to_symbol），可带 evidence 与 confidence

Phase 3：Call Graph（可选高阶）

目标：近似调用关系以辅助理解关键链路，但必须明确置信度。

Python：静态 call graph 精准度有限，先做“显式可解析调用”，并标注 confidence（high/medium/low）

TS：可利用类型信息提升精准度，但工程量更大

输出边类型：calls

4. 输入、输出与 CLI 形态（建议 MVP）
输入参数

repo URL（必填）

branch/commit（可选，默认 main/master）

include/exclude 规则（可选）

max_file_kb（默认 200KB）

是否拆分 RepoBook（按目录分章，默认开启）

是否上传到 NotebookLM（可选，需 notebooklm-py 配置）

输出目录结构（建议）
out/
  RepoBook/
    00_overview.md
    01_src.md
    02_packages_x.md
    03_tests.md
  GraphBook.md
  manifest.json
  graph.json
  stats.json (可选：字数、文件数、跳过原因)
CLI 示例

repo2nlm ingest <repo_url> --branch main --out ./out --max-file-kb 200 --exclude "node_modules/**,dist/**,.git/**"

repo2nlm upload ./out --notebook "<name_or_id>"（内部调用 notebooklm-py）

repo2nlm update <repo_url> --out ./out --upload（增量）

5. 过滤与内容策略（避免爆炸式上下文）
默认排除（建议）

.git/**, node_modules/**, dist/**, build/**, target/**, .venv/**, venv/**

大型锁文件/依赖缓存：pnpm-lock.yaml, package-lock.json, yarn.lock（可只保留摘要）

二进制：图片、音频、视频、压缩包、可执行文件等（只保留元信息）

单文件超过阈值：截断（头/尾）+ 统计 + 保留 raw 链接（可选）

RepoBook 组织原则

保留目录树与路径定位信息

README/docs 优先

入口文件与核心模块优先（见入口检测）

大项目按目录拆章，单章控制在合理大小（例如 50–200KB 或按 token 预算）

6. 入口检测（Entry Points）启发式

用于 GraphBook 的“关键路径”与“核心目录”判断。

Python：__main__.py, main.py, app.py, cli.py, manage.py

JS/TS：src/index.ts, src/main.ts, server.ts, app.ts, bin（package.json）

配置入口：pyproject.toml / setup.cfg / requirements.txt, package.json, tsconfig.json

Web 框架目录：routes/, controllers/, pages/, api/

输出：入口候选列表 + 依据（文件名/配置字段/import 入度等）

7. Import Graph 解析与 resolve 规则（Phase 1）
Python import

解析：

import x, import x as y

from a.b import c

from . import x, from ..foo import bar

resolve：

基于 repo 根与包结构（__init__.py）做模块路径映射

无法映射：标注 external: true

输出边：

type: imports

from: 文件路径

to: 文件路径（内部）或包名（外部）

evidence: 行号 + 原始 import 片段

confidence: 通常 high

JS/TS import

解析：

import ... from '...'

require('...')

import('...')（先做静态字符串参数）

resolve：

相对路径：补全 .ts/.tsx/.js/.jsx 与 index.*

路径别名：Phase1 可忽略或仅做简单 tsconfig paths 支持

npm 包：external

输出同上。

8. 数据格式（必须从 Day1 固化）
manifest.json（示例）
{
  "repo": {"url": "...", "default_branch": "main", "commit": "abc123"},
  "files": [
    {"path": "src/auth.py", "sha": "...", "size": 1234, "lang": "py", "text": true},
    {"path": "src/index.ts", "sha": "...", "size": 4321, "lang": "ts", "text": true}
  ],
  "filters": {"exclude": ["node_modules/**", "dist/**", ".git/**"], "max_file_kb": 200}
}
graph.json（示例）
{
  "edges": [
    {
      "type": "imports",
      "from": "src/client.py",
      "to": "src/auth.py",
      "evidence": {"line": 3, "text": "from . import auth"},
      "confidence": "high",
      "external": false
    },
    {
      "type": "imports",
      "from": "src/app.ts",
      "to": "src/routes/user.ts",
      "evidence": {"line": 12, "text": "import userRoutes from './routes/user'"},
      "confidence": "high",
      "external": false
    }
  ],
  "dirs": [
    {
      "path": "src/routes",
      "role": "路由层（HTTP 接入）",
      "signals": ["目录命名 routes", "被入口 app.ts 引用", "TS 文件占比高"]
    }
  ]
}
9. GraphBook.md 内容模板（固定结构）

项目总览

repo 信息、主要语言、如何运行（从 README/配置提取）

入口文件候选与依据

目录职责地图（树状）

每个目录：role + why(signals)

核心依赖关系（Imports）

Top N 目录/模块（按入度/中心性）

分层推断：基础层/业务层/接口层（基于依赖方向与入口距离）

关系明细（可检索）

Imports：按 from 文件分组，列出 to 与 evidence

（Phase2 后）Exports/Defines/References

（Phase3 后）Calls（含 confidence）

10. RepoBook.md 内容模板（固定结构）

仓库简介（README 摘要）

目录树（完整或裁剪）

章节（按目录拆分）：

目录简介（该目录下文件类型统计 + 入口相关性）

文件列表（路径 + 简短描述/大小）

文件内容（可截断，保留路径标题）

11. 增量更新策略

使用 manifest.json 中的 sha/hash 与当前仓库对比，找出变更文件列表

只对变更文件重做：

import 解析与 resolve

受影响依赖邻居（反向依赖）必要时重算中心性

相关目录职责统计增量更新

RepoBook：只更新涉及的章节或文件段

GraphBook：只更新 imports 明细与目录职责变动部分（或整体重渲染但数据来自增量）

12. 与 NotebookLM 的集成（通过 notebooklm-py）

使用 notebooklm-py 创建/选择 Notebook

上传 Sources：

RepoBook（一个或多个 md）

GraphBook.md

可选：自动发起一组初始化提问（帮助生成学习材料与音频）

“总结系统分层与核心模块”

“给出关键路径：入口→核心业务→持久化/外部依赖”

“列出最重要的 10 个文件及原因”

（注：notebooklm-py 作为非官方 API，具体调用方式以其仓库文档为准。）

13. 非功能需求与边界

规模控制：避免把超大仓库一次性灌入；默认有大小与数量限制，支持选择性导入

可解释性：GraphBook 的每条关系尽量带 evidence（行号/片段）与 confidence

兼容：同时支持 Python 与 JS/TS；monorepo（packages/*）也能工作

安全：私有仓库需要 token；处理时不泄露 token

可配置：include/exclude、最大文件大小、拆章策略、是否生成 calls 等

14. MVP 验收标准（Phase0+Phase1）

对任意公开 repo：

一条命令 ingest 后生成 out/ 目录，包含 RepoBook、GraphBook、manifest、graph

RepoBook 保留目录层级，能定位文件路径

GraphBook 能给出目录职责地图与 imports 关系（内部/外部区分）

上传到 NotebookLM 后，问“这个项目的架构分层与入口是什么？”能得到明显更结构化的回答（相比只上传 repo homepage）
